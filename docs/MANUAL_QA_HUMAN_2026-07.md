# MANUAL_QA_HUMAN 2026-07 — Phase 3 사용자 물리 항목 대화형 런북

> [MANUAL_QA_RUN_2026-07.md](MANUAL_QA_RUN_2026-07.md)(17항목 실행 시트) 중 **adb 로 재현할 수
> 없는 물리 항목**을, 기기 앞 사람 + 로컬 Claude(adb 세션, `C:\dev\ssambership-app`)가
> 한 항목씩 주고받으며 실행하는 런북. 각 항목의 **단계·기대 결과 문구는 시트 원문 그대로**다.
> 화면 조작은 전부 사람이 하고, Claude 는 증적 확보(adb screencap / pull)와 판정만 담당한다
> (실기기에서는 [capture-screenshots.md](capture-screenshots.md)의 에뮬레이터 좌표탭이
> 통하지 않으므로 좌표 탭 내비게이션을 쓰지 않는다).

## 0. 머리말

### 0-1. 기기 슬롯 (시트 0-1 매트릭스의 필수 2대)

| 슬롯 | 기기 | 로그인 계정 | 이 런북 담당 항목 |
|---|---|---|---|
| T1 | Android 태블릿 + 스타일러스(S펜 등) | 멘토 계정(시트 0-3) | B-1, B-2, B-3, C-3(수신측) |
| P1 | Android 폰(가능하면 중저가 1대) | 학생 계정(시트 0-3) | A-4, C-1, C-3(전송측) |

- 빌드·설치·계정 준비는 시트 §0-2·§0-3 을 그대로 따른다. **계정 자격증명·anon key 값은
  이 문서와 증적에 기록하지 않는다.**
- Claude(로컬 세션) 사전 셋업:

```bash
adb devices                    # 연결된 두 기기 시리얼 확인
T1="adb -s <태블릿 시리얼>"    # 예: adb -s R52X...
P1="adb -s <폰 시리얼>"
mkdir -p docs/qa/2026-07       # 증적 저장 폴더(레포 루트 기준)
```

> **셸 전제**: 이 문서의 명령 블록은 전부 **bash 문법**(Windows 에서는 Git Bash 등)이다.
> PowerShell 에서는 `$T1` 변수 패턴이 다르고 `exec-out … > 파일.png` 리다이렉션이 PNG
> 바이너리를 깨뜨린다. PowerShell 로 진행해야 하면 스크린샷은
> `adb -s <serial> shell screencap -p /sdcard/qa-tmp.png` → `adb -s <serial> pull /sdcard/qa-tmp.png docs/qa/2026-07/<항목ID>-<n>.png`
> 2단계 방식으로 대체한다.

### 0-2. 진행 규칙 (한 번에 한 항목)

1. Claude 가 해당 섹션의 **① 사람 수행 절차**를 안내한다.
2. 사람이 기기에서 절차를 수행하고, 지시된 화면에 머문 채 **"완료"**(+관찰 내용)로 응답한다.
3. Claude 가 **③ 증적 확보**를 수행한다(adb screencap / 산출물 pull). 사람은 캡처가 끝날
   때까지 화면을 벗어나지 않는다.
4. Claude 가 **④ 판정 기준**에 따라 판정하고, 결과를 시트(MANUAL_QA_RUN_2026-07.md)의
   통과 체크박스에 반영한다. 실패면 시트 §2 실패 기록 규약대로 실패 로그에 추가한다.
5. 판정까지 끝난 뒤에만 다음 항목으로 넘어간다.

증적 파일명 규약(통과·실패 공통, 시트 §2 와 동일): `docs/qa/2026-07/<항목ID>-<n>.png`
(스크린샷은 `.png`, 기기에서 pull 한 바이너리 산출물은 원본 확장자 유지 — 예: `A-4-2.jpg`)

### 0-4. 조건부 항목 안내 (C-1 · C-3)

> **C-1(한글 IME 조합)·C-3(수식 특수문자)은 조건부 항목이다.** 원래 adb 자동 세션에서
> ADBKeyboard(IME 앱)로 재현하는 항목이며, **ADBKeyboard 설치가 미승인되었거나 조합 중
> 상태/유니코드 입력 재현에 실패한 경우에만** 이 런북으로 이관해 사람이 실기기 키보드로
> 수행한다. 자동 세션에서 이미 통과했다면 여기서는 건너뛴다.
>
> **수행/건너뜀 판단 근거**: 자동(adb) 세션의 항목별 결과는
> [QA_RUN_RESULT_2026-07.md](QA_RUN_RESULT_2026-07.md) **판정 매트릭스**와 시트
> (MANUAL_QA_RUN_2026-07.md)의 체크박스·실패 로그에 기록된다. 두 문서에 해당 항목 기록이
> 없으면 자동 세션 미수행으로 간주하고 이 런북으로 수행한다.

## 1. B-1 필압(의도된 off) — T1

**시트 원문 — 단계**: 첨삭 화면에서 필압을 약→강 바꿔 필기

### ① 사람 수행 절차

사전 상태: T1 에 멘토 계정 로그인. 학생 사진 첨부가 있는 질문 상세에서 첨부의
'첨삭하기'로 **첨삭 화면** 진입. (첨삭 대상 첨부가 없으면 먼저 P1(학생)에서
`/sdcard/Download/qa/qa-img5-annotate.jpg` 를 첨부해 질문을 등록해 둔다 — B-1~B-3 이
같은 첨부를 공유한다.)

1. 첨삭 화면에서 스타일러스로 **필압을 약하게** 하여 한 획을 긋는다.
2. 바로 옆에 **필압을 강하게** 하여 한 획을 긋는다(약→강, 같은 색·같은 도구).
3. 한 획 안에서 약→강으로 힘을 바꿔 가며 긴 획을 하나 더 긋는다.
4. 세 획이 모두 보이는 상태로 화면을 유지한 채 "완료"와 함께 육안 관찰(선폭 변화 여부)을
   응답한다.

### ② 기대 결과 (시트 원문)

선폭 일정(압력→선폭 매핑은 P1 — 변하면 실패)

> 참고: 위 원문의 'P1'은 기기 슬롯이 아니라 **개선 우선순위 P1**(추후 매핑 도입 예정,
> 현재는 의도된 off)을 뜻한다.

### ③ 증적 확보 (Claude)

```bash
$T1 exec-out screencap -p > docs/qa/2026-07/B-1-1.png   # 약·강 획이 모두 보이는 첨삭 화면
```

### ④ 판정 기준·판정 주체

- **1차 판정: 사람**(기기 화면 육안). 필압에 따라 선폭이 변하면 실패.
- Claude 는 `B-1-1.png` 에서 획들의 선폭이 동일한지 재확인(보조 판정)하고 결과를 기록한다.

## 2. B-2 팜 리젝션 — T1 (실손바닥 최종 확인)

> **구분 명시**: 스타일러스 vs 손가락 **이벤트 로직 대조**(adb 로 stylus/finger 모션이벤트를
> 주입해 펜 전용 모드 필터링을 검증)는 **adb 자동 항목**으로 별도 수행된다. 이 섹션은 adb 로
> 재현할 수 없는 **실제 손바닥(대면적 접촉)** 을 사람 손으로 확인하는 최종 물리 항목이다.
> 로직 대조의 수행 여부·결과는 [QA_RUN_RESULT_2026-07.md](QA_RUN_RESULT_2026-07.md) 판정
> 매트릭스에서 확인한다(§0-4 와 같은 규칙). B-2 최종 판정은 로직 대조 + 이 섹션 둘 다
> 끝나야 확정된다.

**시트 원문 — 단계**: 기본(펜 전용) 모드에서 손바닥을 캔버스에 대고 펜 필기

### ① 사람 수행 절차

사전 상태: T1 멘토 로그인, B-1 과 동일하게 첨삭 화면 진입. 입력 모드가 **기본(펜 전용)**
인지 확인(손가락 토글 off).

1. 손바닥을 캔버스 위에 자연스럽게 얹은 상태(글씨 쓰는 자세)로 펜 필기를 한다.
2. 필기 중 손바닥을 캔버스 위에서 문지르듯 움직여 본다 — 손바닥으로 스트로크가 생기거나
   화면이 이동/오입력되는지 관찰한다.
3. **손가락 입력 토글을 켠다.**
4. 손가락으로 캔버스에 그려 본다 — 이번에는 그려져야 한다.
5. 손가락 스트로크가 보이는 화면을 유지한 채 "완료"와 함께 1~2단계 관찰 내용을 응답한다.

### ② 기대 결과 (시트 원문)

손바닥 터치로 스트로크·오입력 없음. 손가락 토글 후엔 손가락도 그려짐

### ③ 증적 확보 (Claude)

```bash
$T1 exec-out screencap -p > docs/qa/2026-07/B-2-1.png   # 펜 전용 모드 필기 결과(손바닥 오입력 없음)
# (사람: 손가락 토글 후 손가락 획까지 그린 화면으로 이동)
$T1 exec-out screencap -p > docs/qa/2026-07/B-2-2.png   # 토글 후 손가락 스트로크
```

### ④ 판정 기준·판정 주체

- **판정 주체: 사람.** 손바닥을 대고 있는 동안의 실시간 거동(오입력 순간)은 캡처로 재구성할
  수 없으므로, 기대 결과 두 문장 모두 사람이 현장에서 판정한다.
- Claude 는 `B-2-1.png` 에 의도치 않은 스트로크가 없는지, `B-2-2.png` 에 손가락 스트로크가
  있는지 보조 확인 후 기록한다.

## 3. B-3 좌표 정합(줌) — T1 (핀치줌 제스처)

> 물리 사유: **핀치줌(두 손가락) 제스처**는 adb 멀티터치 주입으로 신뢰성 있게 재현되지 않아
> 사람이 수행한다.

**시트 원문 — 단계**: 이미지 300%+ 확대 상태에서 특정 글자에 밑줄 → 완료 → 전송본 확인

### ① 사람 수행 절차

사전 상태: T1 멘토 로그인, `qa-img5-annotate.jpg` 첨부(B-1 과 동일) 위 첨삭 화면 진입.

1. 핀치줌으로 이미지의 **L07 행**이 **300% 이상**으로 보이게 확대한다.
2. **L07 행의 문자 `K`**(이미지에 명시된 밑줄 대상) 바로 아래에 밑줄을 한 획 긋는다.
   (다른 글자를 썼다면 어떤 글자인지 응답에 밝힌다.)
3. 이 상태(확대+밑줄, 완료 누르기 전)에서 멈추고 "1차 캡처 요청"이라고 응답한다 → Claude 가
   증적 1을 캡처할 때까지 대기.
4. '완료'를 눌러 첨삭을 저장·전송한다.
5. 채팅/상세에서 **전송본(평탄화본) 첨부를 열어** 해당 글자가 보이게 확대한다.
6. 그 화면을 유지한 채 "완료"와 함께 대상 글자가 무엇인지 응답한다.

### ② 기대 결과 (시트 원문)

평탄화본에서 밑줄이 같은 글자 위(±글자 반폭 이내)

### ③ 증적 확보 (Claude)

```bash
# 3단계(완료 전): 확대 상태의 밑줄 위치
$T1 exec-out screencap -p > docs/qa/2026-07/B-3-1.png
# 6단계: 전송본(평탄화본) 열람 화면
$T1 exec-out screencap -p > docs/qa/2026-07/B-3-2.png
# 필요 시 전송본에서 대상 글자를 더 확대한 추가 컷
$T1 exec-out screencap -p > docs/qa/2026-07/B-3-3.png
```

### ④ 판정 기준·판정 주체

- **판정 주체: Claude.** `B-3-1.png`(첨삭 시점)과 `B-3-2.png`/`B-3-3.png`(평탄화본)를 이미지
  대조하여, **밑줄이 같은 글자 위에 있고 좌우 편차가 ±글자 반폭 이내**인지 판정한다.
- 사람은 수행·화면 유지만 담당한다. Claude 는 판정 근거(대상 글자, 관측 편차)를 시트 기록에
  남긴다.

## 4. A-4 크기 캡 실효 — P1 (카메라 최고화소 세팅·촬영)

**시트 원문 — 단계**: 48MP 급 고해상 촬영 → 첨부 → 업로드

### ① 사람 수행 절차

사전 상태: P1 학생 로그인. **기기 카메라 설정에서 촬영 화소를 최고(48MP/50MP 등 고화소
모드)로 변경**해 둔다.

1. P1 기본 카메라로 세부 묘사가 많은 대상(문제지·인쇄물)을 최고화소로 1장 촬영한다.
2. 앱에서 채팅(또는 IQ 작성) 첨부 시트를 열고, 방금 촬영한 사진을 첨부한다
   (갤러리/파일 소스 — 앱 내 '촬영' 소스로 한 번 더 반복하면 두 경로 모두 확인된다).
3. 업로드가 끝날 때까지 기다린다.
4. 업로드된 첨부를 열어 **확대(crop)** 해 육안 화질을 확인한다.
5. 첨부 열람 화면을 유지한 채 "완료"와 함께 카메라 설정 화소(예: 50MP)와 육안 화질 소감을
   응답한다.

### ② 기대 결과 (시트 원문)

선택 시점 장변 ≤4096·품질85 적용, 그래도 5MB 초과면 장변 2560·JPEG 재인코딩까지 동작해
**최종 업로드 5MB 이하**, 육안 화질 수용 가능

### ③ 증적 확보 (Claude)

```bash
# 업로드 완료된 첨부 열람 화면
$P1 exec-out screencap -p > docs/qa/2026-07/A-4-1.png

# 원본 크기 확인(재인코딩 경로 진입 여부 근거 — 바이트 기록)
$P1 shell ls -l /sdcard/DCIM/Camera/ | tail -5

# 재인코딩 산출물 pull(디버그 빌드 전제 — run-as 가능). 앱 캐시에서 방금 생성된 jpg 를 찾는다.
$P1 shell run-as com.ssambership.ssambership_app sh -c 'ls -lt cache cache/*/ 2>/dev/null | head -20'
$P1 exec-out run-as com.ssambership.ssambership_app cat <캐시의 최신 jpg 경로> \
```
