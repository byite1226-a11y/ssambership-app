# RELEASE_TRIAGE — 감사 40건 '빠른 출시' 분류 (차단 vs 출시 후)

> 목적: `docs/FEATURE_AUDIT.md`의 웹↔앱 차이 목록을 **"지금 출시해도 되는가"** 관점으로 재분류한다.
> 오너가 **[출시 차단]만 먼저 처리하면 출시 가능**하도록 맨 앞에 요약한다.
> 이 문서는 분류·판단 문서일 뿐 수정 지시가 아니다. **이번 작업에서 어떤 `.dart` 코드도, 기존 `FEATURE_AUDIT.md`도 변경하지 않았다.**
> 작성 기준일: 2026-07-02. 작성 주체: S-TRIAGE(읽기·분석 전용).

## 분류 기준

- **[출시 차단]** — 지금 상태로 출시하면 *명백히 잘못된 동작/데이터*가 사용자에게 보이거나, 규칙 위반이 *실제 피해*로 이어짐.
- **[출시 후 OK]** — UX 개선·지엽 규칙·정보 단순화. 없어도 서비스는 정상 작동.
- **[처리중]** — A1·A2는 다른 창에서 구현 중.
- **[설계상 제외]** — CR·IQ·관리자·가입폼·앱내결제·글작성·잔여수표기·메시지수정삭제. 재분류하지 않고 제외 유지.
- **[조건부 차단]** — 서버(RPC/RLS) 규칙 또는 출시 범위에 따라 차단 여부가 갈림. **출시 전 오너가 반드시 확인**해야 하는 항목.

## ★ 출시 차단 요약 (오너가 먼저 볼 것) ★

**결론: 하드(무조건) 출시 차단은 A1·A2 두 건뿐이며, 둘 다 이미 다른 창에서 처리 중이다.**
**감사에서 '높음'으로 잡혔던 D1(커뮤니티 카테고리)은 실제 웹 파일 대조 결과 정합함 — 출시 차단 아님(아래 §D1 확정 참고).**

따라서 오너가 출시 전 실제로 확인/결정할 것은 **처리중 2건 + 조건부 차단 3건**이다.

| 우선 | ID | 기능 | 상태 | 왜 차단인가 | 난이도 |
|---|---|---|---|---|---|
| 1 | **A1** | 질문 과목 = 멘토 지정 과목만 | 🔧 처리중 | 학생이 멘토 미담당 과목으로 질문을 태깅 → 잘못된 데이터가 멘토에게 전달 | 보통 |

| 5 | **D9** | 숏폼 영상 재생 | ⚠️ 조건부 | 재생 플러그인 미도입(썸네일만) → 숏폼 영상을 **출시 범위에 포함하면** 재생 안 되는 영상이 노출 | 어려움 |

**조건부 3건 처리 지침(코드 미변경, 오너 결정용):**
- **B3**: `mentor_directory_list_v2` RPC 정의에서 `approved/verified/active` + `teaching_subjects>0` 필터가 서버측에 있는지 확인. 있으면 → **출시 후 OK로 강등**. 없으면 → 앱에 클라이언트 필터 추가(쉬움) 후 출시.
- **D8**: `shortform_reactions` 테이블에 `scrap` 종류 저장이 유효한지 확인(§9-8). 무효면 → 앱 숏폼 상세에서 scrap 버튼을 **숨김**(출시 후 재도입). 유효면 출시 후 OK.
- **D9**: 숏폼 영상 재생을 이번 출시 범위에 넣을지 결정. 안 넣으면 → 숏폼 탭/영상 진입을 가리거나 "준비 중" 처리. (S6 인프라 인수인계 대기 항목이므로 범위 제외가 현실적.)

## D1 확정 — 커뮤니티 카테고리 표준 (웹 코드 직접 대조)

**감사 문서 [D1]의 "웹 all ↔ 앱 free 매핑 상이" 판정은 오독이었다.** 감사 당시 웹 목록을 `all/study/school/career/college`(5개)로 적었으나, 실제 웹 파일에는 `free`가 포함된 **6개**다. `all`은 저장값이 아니라 **UI 전용 '전체' 칩**이다.

### 웹 실제 카테고리 집합 (`lib/community/communityBoardConstants.ts`, 읽기 전용)

```
COMMUNITY_POST_CATEGORIES = [ all(전체), study(학습법), school(내신), career(진로), college(대학생활), free(자유) ]
```

- **`all`은 UI 전용** — `CommunityPostCategoryValue = Exclude<슬러그, "all">` 로 타입에서 제외.
- **실제 저장값 집합** = `COMMUNITY_POST_CATEGORY_SLUGS` = `all` 제외 = **`{ study, school, career, college, free }`** (5개).
- 미분류 입력은 `normalizeCommunityPostCategory()`가 **`free`로 정규화**(웹 기본값 = free).

### 앱 카테고리 집합 (`lib/features/community/data/community_labels.dart`, 읽기 전용)

```
{ study(학습), school(내신), free(자유), college(대학생활), career(진로) }  // 순서만 다름
```
- 주석대로 `'전체'`는 화면에서 앞에 별도로 붙임 → 앱도 `all`을 저장값으로 쓰지 않음(웹과 동일 구조).

### 확정 결과

| 항목 | 웹(저장값) | 앱 | 판정 |
|---|---|---|---|
| 저장 코드 집합 | `study, school, career, college, free` | `study, school, free, college, career` | **동일**(순서만 상이) |
| `all`(전체) | UI 전용 칩 | UI에서 별도 부착 | **동일 구조** |
| 정규화 기본값 | `free` | (앱은 미정 코드 → '기타' 폴백) | 저장 경로 없음(앱은 글작성 미제공)이라 무관 |
| 라벨 텍스트 | study=**학습법** | study=**학습** | 소소한 표현차(코드 무관) |

**→ D1은 필터가 깨지는 정합성 위험이 아니다.** 앱과 웹의 **저장 category 코드가 이미 일치**하므로 같은 데이터가 양쪽에서 동일하게 필터·표시된다. 커뮤니티 글 작성은 웹 전용(§8)이라 저장값을 만드는 쪽은 웹뿐이고, 앱은 그 5개 코드를 전부 정확히 매핑한다.

**앱이 (선택적으로) 맞추면 좋은 것 — 코드 수정은 하지 않음, 문서 권고만:**
1. **라벨 텍스트 통일(선택)**: `community_labels.dart`의 `study` 라벨을 웹과 동일하게 `'학습'` → `'학습법'`으로 맞추면 완전 일치. (기능 영향 없음, 순수 텍스트.)
2. **정렬 순서(선택)**: 필터칩 노출 순서를 웹(`study·school·career·college·free`)과 맞추면 사용자 인지 일관성↑. (동작 영향 없음.)
3. **숏폼 카테고리(참고)**: 웹 숏폼(`communityShortformConstants.ts`)의 저장값은 `{ study, school, career, college }` (게시판과 달리 `free` 없음, `all`은 UI 전용). 앱은 숏폼 카테고리 필터 자체가 없음(D4) → 코드 불일치 아님(필터 미도입일 뿐).
